# 🌐 Day 35: Exposing Serverless Backends with Amazon API Gateway

## 📋 Project Overview
Running a serverless AWS Lambda function in the console is great, but real-world applications require public-facing endpoints. Today, I built the "front door" for my serverless architecture using **Amazon API Gateway**, allowing external clients (like web browsers or frontend apps) to trigger my Python backend.

## ⚙️ Core Concepts Explored
* **API Gateway:** A fully managed service that makes it easy to create, publish, maintain, monitor, and secure APIs at any scale.
* **HTTP APIs vs. REST APIs:** Chose the HTTP API protocol for its lower latency, cost-effectiveness, and modern features, making it ideal for serverless integrations.
* **Routes & Methods:** Configured a `GET` method on the root route (`/`) to properly handle incoming browser requests.
* **Lambda Integration:** Securely bridged the API Gateway with the existing Lambda function, granting the gateway permission to invoke the code.

## 🛠️ Steps Executed
1. **API Provisioning:** Created a new HTTP API in the AWS Management Console.
2. **Lambda Mapping:** Configured an integration targeting my `MyFirstDevOpsFunction` (Python 3.12).
3. **Route Configuration:** Defined a `GET` route to map incoming HTTP requests directly to the Lambda backend.
4. **Deployment & Invocation:** Deployed the API to the `$default` stage and retrieved the public **Invoke URL**.
5. **Validation:** Successfully hit the Invoke URL via a standard web browser and received the `Hello Cloud Ninja!` JSON response generated by the Python script.
